# Topic Intake Workflow

새 학습 주제를 모듈로 만들기 전에 확인합니다.

## 확인 항목

- 이 주제는 어떤 트랙에 속하는가?
- 이 주제는 기본 `code-first` 흐름으로 충분한가, 아니면 보조 경로로 `theory-first`가 필요한가?
- 오늘 1-3시간 안에 끝낼 수 있는가?
- 실습 산출물은 무엇인가?
- 이론 중심 주제라면 개념 산출물은 무엇인가?
- 선행 개념은 무엇인가?
- 위협 모델, 개념 지도, 용어 경계처럼 먼저 고정해야 할 이론 구조가 있는가?
- 기존 로드맵이나 약점과 연결되는가?
- reference 코드는 언제 보여줄 것인가?

## 이론 중심 주제 보조 경로

기본 학습 방식은 여전히 `code-first`입니다. 다만 네트워크, 보안, 운영체제, 분산 시스템처럼 이론 비중이 큰 주제는 기존 흐름을 대체하지 않고 `theory-first` 보조 경로를 선택할 수 있습니다.

이 경우 `practice/` 코드는 학습의 주 산출물이 아니라 관찰과 검산 도구가 될 수 있습니다. 대신 아래 중 최소 하나를 산출물로 남깁니다.

- 위협 모델 또는 문제 상황
- 핵심 개념 지도
- 헷갈리기 쉬운 용어 경계
- 관찰 실험 결과
- 자신의 말로 쓴 설명과 반례

단, 문서만 남기고 끝내지 않습니다. 가능하면 작은 재현 코드, 브라우저/CLI 관찰, 로그, 테스트, 다이어그램 중 하나로 이론을 검산합니다. 이 보조 경로는 `practice/`, `reference/`, `notes.md` 구조를 유지합니다.

## 모듈 생성 기준

새 모듈은 아래 구조를 따릅니다.

```text
learning/<track>/<area>/<NN-name>/
├── README.md
├── guide.md
├── practice/
├── reference/
├── notes.md
└── requirements.txt
```

`requirements.txt`는 필요한 경우에만 둡니다.
